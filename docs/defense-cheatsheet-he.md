# חלק א' — מה בנינו וכל הכלים (עמוד 1)

## 1. מה זה הפרויקט (משפט פתיחה)
אפליקציית מובייל + שרת לאיתור שירותים יהודיים בכל העולם — מסעדות כשרות, בתי כנסת, מנייני תפילה ואירוח שבת — עם **חיפוש חכם מבוסס AI** ו**חיפוש לפי מיקום GPS**. פלטפורמה אחת שמרכזת מידע שהיום מפוזר בעשרות מקורות.

## 2. ארכיטקטורה כללית
- **Mobile:** React Native + Expo + TypeScript (אפליקציה אחת ל-iOS ו-Android)
- **Backend:** NestJS (Node.js) + TypeScript — REST API
- **DB:** PostgreSQL + תוסף PostGIS (חישובי מרחק גאוגרפיים) + TypeORM (ORM)
- **Real-time:** Socket.IO (צ'אט ואירוח)
- **תמונות:** Cloudinary
- **פריסה:** שרת VPS, PM2 לניהול התהליך

**זרימה:** מובייל → קורא ל-REST API של NestJS → NestJS פונה ל-PostgreSQL דרך TypeORM → מחזיר JSON → מובייל מציג.

## 3. מודל ה-AI (הלב של הפרויקט — להכיר הכי טוב)
- **נבנה מאפס** — בלי ChatGPT, בלי ספריות ML (sklearn וכו'). אומן ב-Python ויושם מחדש ב-TypeScript בתוך השרת.
- **שני מודלים:**
  1. **סיווג כוונה (Intent):** restaurant / synagogue / minyan / hosting
  2. **סיווג נוסח תפילה:** אשכנז / ספרד / חב"ד / תימני
- **+ שכבת חוקים (rules)** מעל ה-ML — מחלצת כשרות, סוג אוכל, מיקום ונוסח מתוך השאילתה.
- **האלגוריתם:**
  - **TF-IDF** — ממיר טקסט לווקטור מספרי לפי חשיבות מילים (מילה נדירה = משקל גבוה). משתמש ב-unigrams + bigrams.
  - **Naive Bayes** — מחשב לכל קטגוריה הסתברות = prior × מכפלת הסבירויות של המילים. בוחר את הגבוהה ביותר. עובד ב-log-space + החלקת Laplace.
- **הערכה:** חלוקת Train/Validation/Test 70/15/15 (seed=42). דיוק held-out: **82.6% Test, 85.3% Validation**. מדדים: Precision / Recall / F1 לכל קטגוריה.
- **למה הגישה הזו:** רצה מקומית בשרת, מהירה, ללא תלות בשירות חיצוני ובלי עלות, וקלה להסבר.

## 4. חיפוש לפי מיקום (GPS / PostGIS)
- **PostGIS** מאפשר שאילתות גאוגרפיות — מחשב מרחק אמיתי בין נקודות.
- חיפוש לפי GPS של המשתמש, לפי עיר, ולפי מדינה, עם מרחק מוגדר במטרים.

# חלק א' (המשך) — אבטחה, עומסים, נתונים (עמוד 2)

## 5. אבטחה (המנגנונים — כולם בקוד, אמיתיים)
**אימות והרשאות:**
- **JWT** — טוקן גישה חתום לכל בקשה מוגנת.
- **Refresh Token** — נוצר אקראית (64 בייט), נשמר ב-DB מגובב ב-SHA-256, תוקף 7 ימים, מבוטל ב-logout.
- **Bcrypt** (10 rounds + salt) — סיסמאות אף פעם לא נשמרות גלויות.
- **Guards + RBAC** — `JwtAuthGuard` בודק טוקן, `AdminGuard` בודק תפקיד מול ה-DB.

**הגנה מפני איומים:**
- **SQL Injection** ← Parameterized Queries דרך TypeORM (אף פעם לא בונים מחרוזת SQL ידנית).
- **קלט זדוני** ← DTO Validation עם `whitelist` + `forbidNonWhitelisted` (זורק שדות לא מוכרים).
- **Brute-force / DoS** ← Rate Limiting (Throttler): 500 בקשות/דקה גלובלי, 5/דקה ל-login, 3/שעה ליצירת קשר.
- **כותרות אבטחה:** HSTS, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy.
- **CORS Whitelist** — בפרודקשן רק מקורות מורשים.
- **HTTPS** נכפה בפרודקשן (redirect מ-http).
- **Audit Logging** — תיעוד פעולות רגישות.
- עיקרון: **Defense in Depth** — כמה שכבות הגנה, לא נקודת כשל יחידה.

## 6. בדיקת עומסים (k6)
- כלי: **k6**, 60 משתמשים וירטואליים מקבילים, 4 תרחישים (health, destinations, restaurants, search).
- **התוצאה החשובה:** נראה "63% כישלון" — **זה לא קריסה, זה ה-Rate Limiter עובד.** בכל תרחיש בדיוק 500 בקשות הצליחו ואז השאר קיבלו HTTP 429. הבדיקה רצה מ-IP אחד, וההגנה חוסמת 500/דקה.
- ביצועים אמיתיים: זמן תגובה חציוני **~73-79ms** גם תחת עומס. p95 גבוה יותר בגלל הבקשות שנחסמו.
- **המסקנה שמציגים:** המערכת יציבה תחת עומס, והגנת ה-DoS הוכיחה את עצמה.

## 7. איסוף מאגר הנתונים (חשוב למחוון — "איסוף מעניין, לא יצירה ידנית")
- **לא הזנו ידנית.** בנינו **pipeline אוטומטי לאיסוף נתונים (scraping)**.
- **מקורות:** אתר הרבנות הראשית, מועצות דתיות עירוניות, Google Maps.
- **כלי:** **Playwright** (דפדפן Chromium אוטומטי) — כי האתרים משתמשים ב-JavaScript לדפדוף, ו-scraping פשוט (requests/BeautifulSoup) לא מספיק.
- **שלבי ה-pipeline:** Scrape → ניקוי ונרמול → JSON מובנה → **Geocoding** (המרת כתובת לקואורדינטות דרך Nominatim) → ייבוא מרוכז ל-DB.
- **מניעת כפילויות:** Set בזיכרון + UNIQUE INDEX ב-DB.
- **למה לא Google Places API:** נבחן ונדחה — יקר מדי לכמות הנתונים.

## 8. בדיקות ואיכות
- בדיקות יחידה ב-Backend (Jest) — אימות, מנייני, אירוח ועוד.
- בדיקות למודל ה-AI ב-Python (`test_classifier.py`) — בודקות דיוק held-out + סיווג נכון לכל קטגוריה.

# חלק ב' — שאלות ותשובות (עמוד 3)

**ש: זה באמת AI או שזה סתם מילות מפתח?**
ת: זה מודל למידת מכונה אמיתי — TF-IDF להמרת טקסט לווקטור + Naive Bayes לסיווג הסתברותי. בנינו אותו מאפס, אימנו על דאטה אמיתי, ומדדנו דיוק על נתונים שהמודל לא ראה (82.6%). זה לא חיפוש מחרוזת — הוא מבין משפט חופשי ומפרק אותו לכמה פרמטרים.

**ש: למה לא השתמשתם ב-ChatGPT / API מוכן?**
ת: רצינו מודל שרץ מקומית, מהיר, בלי עלות ובלי תלות בשירות חיצוני, ושנבין ונשלוט בו לגמרי. בנייה מאפס גם הדגימה את ההבנה שלנו באלגוריתמים.

**ש: איך מדדתם את הדיוק? 82% זה מה?**
ת: חילקנו את הדאטה ל-Train/Validation/Test ביחס 70/15/15. אימנו על ה-Train ומדדנו על ה-Test שהמודל מעולם לא ראה — כך שזה דיוק כללי אמיתי, לא שינון. בדקנו Precision, Recall ו-F1 לכל קטגוריה.

**ש: מה הקטגוריה החלשה ביותר ולמה?**
ת: minyan (recall ~0.71) — כי יש פחות דוגמאות אימון ושכיחות חפיפה בניסוח עם synagogue. בגלל זה יש שכבת חוקים משלימה ו-fallback למקרי ספק.

**ש: מה קורה אם המודל טועה / לא בטוח?**
ת: יש fallback — אם הביטחון נמוך, שכבת החוקים משלימה (מילות מפתח לכשרות, נוסח, מיקום), והמשתמש תמיד יכול לסנן ידנית.

**ש: התמיכה באנגלית?**
ת: יש תמיכה ראשונית באנגלית, אבל המודל מותאם בעיקר לעברית כי שם רוב הדאטה. זו נקודה לשיפור עתידי.

**ש: איך אספתם את הנתונים? ידנית?**
ת: לא — בנינו pipeline אוטומטי עם Playwright ש-scrape-ת אתרי רבנות ומועצות דתיות, מנרמל את הנתונים ל-JSON, מבצע geocoding לכתובות וטוען ל-DB. זה תהליך הנדסי, לא הקלדה ידנית.

**ש: איך מנעתם כפילויות בנתונים?**
ת: Set בזיכרון בזמן האיסוף + UNIQUE INDEX ב-DB שמונע הכנסת רשומה כפולה.

# חלק ב' (המשך) — שו"ת אבטחה ומערכת (עמוד 4)

**ש: איך מאובטחות הסיסמאות?**
ת: Bcrypt עם 10 rounds ו-salt. הסיסמה לעולם לא נשמרת גלויה — רק ה-hash. גם אם ה-DB ידלוף, אי אפשר לשחזר סיסמאות.

**ש: מה ההבדל בין JWT ל-Refresh Token אצלכם?**
ת: ה-JWT הוא טוקן גישה קצר-טווח שנשלח בכל בקשה. ה-Refresh Token ארוך-טווח (7 ימים), נשמר ב-DB מגובב ב-SHA-256, ומשמש לחידוש ה-JWT בלי להתחבר מחדש. ב-logout הוא מבוטל.

**ש: איך אתם מוגנים מ-SQL Injection?**
ת: עובדים רק דרך TypeORM עם Parameterized Queries — אף פעם לא בונים מחרוזת SQL ידנית עם קלט משתמש. בנוסף, DTO Validation מסנן קלט לא חוקי.

**ש: מה זה ה-Rate Limiting ולמה?**
ת: מגביל כמות בקשות לכל IP — 500/דקה כללי, 5/דקה ל-login. מונע התקפות brute-force על סיסמאות ו-DoS שמציף את השרת.

**ש: בבדיקת העומס 63% נכשלו — המערכת לא עומדת בעומס?**
ת: להפך. אלו לא שגיאות שרת — אלו בקשות שה-Rate Limiter חסם בכוונה (HTTP 429). הבדיקה רצה מ-IP אחד, וההגנה חוסמת 500 בקשות לדקה, אז בדיוק 500 עברו בכל תרחיש. זמן התגובה החציוני נשאר ~79ms. זה למעשה מוכיח שהגנת ה-DoS עובדת.

**ש: למה PostGIS ולא חישוב מרחק רגיל?**
ת: PostGIS מבצע חישובי מרחק גאוגרפיים מדויקים (על פני כדור הארץ) ברמת ה-DB, יעיל ומהיר יותר מחישוב ידני בקוד, ותומך בשאילתות "מצא הכל ברדיוס X".

**ש: למה NestJS ולא Express רגיל?**
ת: NestJS נותן מבנה מודולרי מסודר, הזרקת תלויות, Guards/Pipes מובנים לאבטחה ו-validation, ותמיכה מצוינת ב-TypeScript. זה הופך את הקוד לתחזוקתי וניתן לבדיקה.

**ש: מה היו האתגרים הגדולים?**
ת: (1) בניית מודל ה-AI מאפס והבאתו לדיוק טוב; (2) Geocoding של אלפי כתובות בלי לחרוג ממגבלות ה-API; (3) חיפוש גאוגרפי יעיל; (4) אבטחה רב-שכבתית.

**ש: מה הייתם משפרים בהמשך?**
ת: שיפור דיוק האנגלית במודל, הרחבת מאגר הערים, התראות בזמן אמת על מנייני, והמלצות מותאמות אישית.

## 3 הדברים שחייבים לזכור בעל פה
1. **ה-AI נבנה מאפס** (TF-IDF + Naive Bayes), נמדד נכון (82.6% held-out) — זה הבידול.
2. **ה-63% בעומס = ה-Rate Limiter עובד**, לא קריסה.
3. **הנתונים נאספו ב-pipeline אוטומטי (Playwright)**, לא ידנית.
