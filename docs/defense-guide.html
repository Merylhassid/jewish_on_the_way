1. תקציר הפרויקט וארכיטקטורה

תקציר

הפרויקט הוא אפליקציה בשם Jewish On The Way, שמטרתה לעזור ליהודים ומטיילים יהודים למצוא שירותים יהודיים לפי מיקום ויעד: מסעדות כשרות, בתי כנסת, מניינים, אירוח שבת, ויעדים רלוונטיים בעולם. הרעיון המרכזי הוא שהמשתמש לא צריך לדעת בדיוק איפה לחפש או באיזו קטגוריה לבחור — הוא כותב חיפוש חופשי כמו "פיצה במזכרת בתיה", "בית כנסת קרוב", "מניין שחרית ספרדי בירושלים" או "אירוח שבת בפוקט", והמערכת מבינה את הכוונה, מזהה את הקטגוריה, מזהה יעד אם קיים, ומנווטת למסך המתאים עם תוצאות רלוונטיות.

שלוש שכבות (3-Tier)

המערכת בנויה בארכיטקטורת 3-tier קלאסית עם הפרדת אחריות מלאה:

  • Mobile (Client) — אפליקציית React Native + Expo. רצה על הטלפון, אחראית רק על תצוגה ואינטראקציה. לא מכילה לוגיקה עסקית או גישה ישירה ל-DB.
  • Backend (Server) — שרת NestJS על Node.js, פורט 3000 בפרודקשן (3001 בפיתוח). מכיל את כל הלוגיקה: אימות, חיפוש חכם, שאילתות, הרשאות.
  • Database — PostgreSQL ענן (Neon) עם תוסף PostGIS לחישובים גאוגרפיים.

איך הם מדברים

  • המובייל מדבר עם הבקאנד דרך REST API מעל HTTP, בפורמט JSON, באמצעות ספריית Axios.
  • הבקאנד מדבר עם ה-DB דרך TypeORM (ORM — ממפה טבלאות לאובייקטים).
  • הפרדה מוחלטת: המובייל אף פעם לא ניגש ל-DB ישירות. כל בקשה עוברת דרך ה-API. זה עיקרון אבטחה ותחזוקה בסיסי.

Flow מלא של חיפוש (מהקלדה ועד תוצאות)

  1. משתמש מקליד "פיצה במזכרת בתיה" במסך הבית.
  2. המובייל שולח POST /search { text, lat?, lng? } (Axios + JWT בכותרת).
  3. הבקאנד: (א) ClassifierService מסווג קטגוריה → restaurant. (ב) שכבת חוקים בודקת מילות אוכל → מאשר restaurant. (ג) DestinationIndexService מזהה "מזכרת בתיה" → destinationId=290. (ד) בונה route: /restaurants/290?q=פיצה (בלי useUserGps, כי יעד מפורש).
  4. הבקאנד מחזיר JSON עם ה-route.
  5. המובייל מנווט (Expo Router) למסך המסעדות עם הפרמטרים.
  6. המסך שולח GET /restaurants/search?destinationId=290&q=פיצה.
  7. smartSearch מריץ שאילתת PostGIS, מחזיר עד 50 תוצאות לפי מרחק מהיעד.
  8. המסך מציג רשימה עם מרחקים.
הארכיטקטורה היא 3-tier עם הפרדת אחריות מלאה: המובייל אחראי על UI, הבקאנד על כל הלוגיקה והאבטחה, וה-DB על אחסון. התקשורת היא REST/JSON — וזה מאפשר שגם אם נחליף את המובייל בעתיד, אותו בקאנד ישרת אותו.

3. החיפוש החכם — צלילה עמוקה (הלב של הפרויקט)

מה הבעיה שפתרנו

המשתמש כותב טקסט חופשי בעברית או אנגלית — "מסעדה כשרה בפריז", "מניין שחרית מחר", "אירוח לשבת בתל אביב". המערכת צריכה: (1) להבין מה הוא רוצה (מסעדה? בית כנסת? מניין? אירוח?); (2) להבין איפה (איזו עיר/מדינה); (3) לנווט אותו למסך הנכון עם הפילטרים הנכונים.

החלק של "להבין מה הוא רוצה" — זה ה-AI. זו בעיית סיווג טקסט (text classification): מקבלים משפט, פולטים תווית מתוך 4 אפשרויות. בדיוק כמו מסנן spam שמסווג מייל ל"זבל/תקין", רק עם 4 קטגוריות.

הבחנה קריטית — פייפליין דו-שלבי

  • שלב ההבנה (AI) — מודל ML שמסווג כוונה.
  • שלב הביצוע (דטרמיניסטי) — שאילתות SQL/PostGIS שמביאות תוצאות.
ה-AI מבין את הכוונה; הקוד הדטרמיניסטי מביא את התוצאות. AI איפה שצריך גמישות בשפה, וקוד מדויק איפה שצריך עקביות.

הפייפליין המלא — POST /search

הקובץ המרכזי: backend/src/ai/search.controller.ts, מתודה search().

  1. normalizeTypos — תיקון שגיאות כתיב בטוחות (פיצמ→פיצה).
  2. Model 1 (ClassifierService) → קטגוריה + confidence (restaurant / synagogue / minyan / hosting).
  3. שכבת overrides (אוכל / בית כנסת / אירוח).
  4. אם confidence < 0.45 ואין override: אולי זה שם עיר בלבד? אם כן → ניווט ליעד; אחרת → "לא הבנתי, נסה שוב".
  5. Model 2 (נוסח) — רק לבית כנסת/מניין.
  6. extractRestaurantFilters — סוג + כשרות (regex, לא ML).
  7. זיהוי יעד: טקסט מפורש → מדינה → fuzzy → GPS הקרוב.
  8. getRoute — בונה URL עם הדגלים (expandNearby, useUserGps).
  9. שמירת SearchFeedback ל-DB (ללמידה עתידית).

4. המודל מאפס — TF-IDF, Naive Bayes, אימון וייצוא

למה בנינו מודל מאפס ולא ChatGPT / ספרייה (קריטי!)

זו השאלה מספר 1 שישאלו. חמש סיבות:

  1. דרישה אקדמית — הפרויקט דורש שנבנה ונבין מודל ML. בנינו את TF-IDF ואת Naive Bayes ביד, בלי scikit-learn, בלי TensorFlow. כל שורת מתמטיקה היא שלנו.
  2. עלות — אפס. אין תשלום לכל קריאה (ChatGPT עולה כסף לכל בקשה).
  3. פרטיות — הטקסט של המשתמש לא יוצא מהשרת שלנו לחברה חיצונית.
  4. מהירות — סיווג תוך מילישניות מקומית, בלי latency של קריאת רשת לשרת חיצוני.
  5. דטרמיניזם — אותו קלט תמיד נותן אותו פלט. LLM יכול לתת תשובות שונות; אצלנו זה צפוי ובדיק.
בנינו את המודל מאפס ב-Python — TF-IDF ו-Naive Bayes בלי שום ספריית ML. זו דרישה אקדמית, אבל גם החלטה נכונה: חינם, פרטי, מהיר, ודטרמיניסטי. לבעיה של סיווג טקסט קצר ל-4 קטגוריות, מודל קלאסי כזה מנצח LLM ענק מכל בחינה מעשית.

הנתונים (Dataset)

  • data.csv — 728 משפטים מתויגים ידנית, כל שורה: טקסט, תווית.
  • התפלגות הקטגוריות: minyan 221, synagogue 187, restaurant 175, hosting 145. מאוזן יחסית (אין קטגוריה שמשתלטת).
  • מודל הנוסח: data_denomination.csv — 143 משפטים, 4 נוסחים.
  • אוצר מילים (vocabulary): 744 מילים ייחודיות.
  • כולל עברית ואנגלית מעורבבות, כדי שהמודל יעבוד בשתי השפות.

צעד 1 — Tokenization (פירוק למילים)

פונקציה tokenize (זהה ב-Python וב-TypeScript): מורידה לאותיות קטנות, מנרמלת חב"ד (עם גרשיים) → חבד כמילה אחת, ושומרת רק מילים עברית/אנגלית (Regex [א-ת]+|[a-z]+), זורקת פיסוק ומספרים.

"מסעדה כשרה בפריז" → ["מסעדה", "כשרה", "בפריז"]

צעד 2 — Features: מילים + Bigrams

פונקציה generate_features מוסיפה גם זוגות מילים רצופים (bigrams), מופרדים ב-__. למה: מילה בודדת מאבדת הקשר. "מניין שחרית" כזוג חזק יותר ממילים נפרדות; "בית" לבד לא אומר בית כנסת, אבל "בית__כנסת" כן. כדי למנוע overfitting, bigram נשמר רק אם הופיע ב-≥3 משפטים (MIN_BIGRAM_DF).

צעד 3 — TF-IDF (הופך מילים למספרים)

המחשב לא מבין מילים, רק מספרים. TF-IDF הופך כל משפט לוקטור מספרי.

  • TF (Term Frequency): כמה פעמים פיצ'ר מופיע במשפט, חלקי מספר הפיצ'רים (tf += 1.0/len(features)). מנרמל לפי אורך — משפט ארוך לא מקבל יתרון.
  • IDF (Inverse Document Frequency): כמה המילה נדירה בכל הדאטה. הנוסחה: idf = log(מספר_משפטים / כמה_מכילים_את_המילה). "כשר" מופיעה כמעט בכל משפטי מסעדה → IDF נמוך (לא מבדילה). "מניין" מופיעה רק במניין → IDF גבוה (מילה "חזקה").
  • TF-IDF = TF × IDF. התוצאה: וקטור שבו המילים החשובות והמבדילות בולטות, והשכיחות נחלשות.
TF-IDF נותן משקל למילים. מילה שמופיעה בכל המשפטים כמו 'אני' או 'רוצה' כמעט לא נחשבת. מילה ייחודית כמו 'מניין' מקבלת משקל גבוה — היא מסגירה את הכוונה.

צעד 4 — Naive Bayes (הסיווג עצמו)

מבוסס על משפט בייס: P(קטגוריה|משפט) ∝ P(קטגוריה) × P(משפט|קטגוריה). בפשטות: "אם המשפט שייך לקטגוריה X, כמה הגיוני שייראה ככה?" — בודקים לכל 4 הקטגוריות ובוחרים את הכי הגיונית.

שני רכיבים שהמודל לומד באימון (fit):

  • Prior (class_probs): כמה הקטגוריה שכיחה מראש. אם 175/728 הם מסעדה → prior = 0.24.
  • Likelihood (feature_probs): ממוצע ערך ה-TF-IDF של כל פיצ'ר בתוך כל קטגוריה.

הניבוי (predict_one): לכל קטגוריה מחשבים ציון score = log(prior) + Σ [tfidf × log(feature_prob)], ובוחרים את הגבוה.

שלוש דקויות שצריך לדעת להסביר:

  1. למה "Naive" (תמים)? כי מניחים שכל המילים בלתי-תלויות זו בזו. הנחה לא מדויקת, אבל עובדת מצוין לסיווג טקסט ומפשטת מאוד את החישוב.
  2. למה log? כי מכפילים הרבה הסתברויות קטנות → המספר נעשה זעיר מדי והמחשב מאבד דיוק (underflow). לוגריתם הופך כפל לחיבור ופותר את זה.
  3. למה +1e-9 (smoothing)? אם feature_prob = 0 אז log(0) = מינוס אינסוף = קריסה. מוסיפים מספר זעיר. זו גרסה של Laplace smoothing.

צעד 5 — Softmax (מ"ציון" ל"ביטחון")

ציוני Naive Bayes הם לוגריתמים שליליים — לא קריאים. Softmax הופך אותם להסתברויות שמסתכמות ל-1:

{ restaurant: 0.92, synagogue: 0.05, minyan: 0.02, hosting: 0.01 }

ה-confidence הוא ההסתברות של המנצח (0.92). אם הוא < 0.45 — המודל "לא בטוח", ואנחנו לא מנחשים אלא מבקשים מהמשתמש לנסח מחדש.

Softmax הופך את הציונים הגולמיים להסתברויות. אם הביטחון נמוך מ-45% — אנחנו עוצרים ואומרים 'לא הבנתי', במקום לנחש ולהטעות.

תהליך האימון (Pipeline) — ולמה הוא נכון מתודולוגית (נקודת חוזק!)

הקוד (classifier.py) עושה הכל לפי הספר:

  1. טעינה — 728 משפטים.
  2. פיצול 70/15/15 (three_way_split, seed=42): Train (70%) — המודל לומד; Validation (15%) — לכוונון; Test (15%) — נחסם, נוגעים פעם אחת בסוף.
  3. TF-IDF נלמד רק מ-Train — קריטי! מונע data leakage (דליפת מידע מהמבחן לאימון).
  4. אימון Naive Bayes על Train בלבד.
  5. הערכה על Validation — מותר לשפר לפי זה.
  6. ציון סופי על Test — פעם אחת. הקוד מדפיס: "זהו הציון האמיתי — ללא הטיה".

מדדי הערכה (לא רק accuracy): Accuracy (אחוז ניחושים נכונים), Precision (מתוך מה שניחשנו "מסעדה", כמה באמת מסעדה), Recall (מתוך כל המסעדות האמיתיות, כמה תפסנו), F1 (ממוצע הרמוני של Precision ו-Recall).

דקות חשובה לכנות: הערכת הדיוק נעשית על split, אבל המודל המיוצא לפרודקשן (export_model.py) מאומן על 100% מהדאטה. זו פרקטיקה תקנית — מודדים הכללה על held-out, אבל המודל הסופי משתמש בכל דוגמה זמינה כדי ללמוד מקסימום.

אימנו לפי המתודולוגיה הנכונה: פיצול 70/15/15, TF-IDF שנלמד רק מ-Train כדי למנוע data leakage, ו-test set שנגענו בו פעם אחת. מדדנו Precision, Recall ו-F1 — לא רק accuracy. הציון שדיווחנו הוא held-out — מדד הכללה אמיתי, לא שינון.

הגשר Python → TypeScript

הבעיה: אימנו ב-Python, אבל השרת רץ ב-TypeScript (NestJS). הפתרון: export_model.py שומר את כל מה שהמודל למד ל-model.json (vocab, idf, class_probs, feature_probs, classes). ב-TypeScript, ClassifierService טוען את ה-JSON בעליית השרת (onModuleInit) ומיישם את אותו אלגוריתם בדיוק (tokenize → TF-IDF → Naive Bayes → softmax).

אימנו ב-Python, אבל השרת ב-TypeScript. ייצאנו את המודל ל-JSON — אוצר המילים, ערכי ה-IDF וההסתברויות — וטענו אותו בשרת שמיישם את אותו אלגוריתם. ה'מוח' זהה, רק השפה השתנתה. היתרון: אין צורך בשרת Python נפרד; המודל רץ בתוך הבקאנד.

ההבדל בין Training ל-Inference

Training = שלב הלמידה: לוקחים דאטה, מחשבים TF-IDF, לומדים הסתברויות, שומרים מודל (בתיקיית ai-model). Inference = שלב השימוש: משתמש כותב משפט חדש, השרת טוען את המודל הקיים ומחזיר קטגוריה (בתוך backend/src/ai).

המודל השני — נוסח (Denomination)

DenominationClassifierService — אותו אלגוריתם בדיוק (TF-IDF + NB + Softmax), מודל נפרד (denomination_model.json), רץ רק כשהקטגוריה היא בית כנסת/מניין. 4 נוסחים: אשכנז / ספרד / חב"ד / תימני. סף ביטחון 0.45 — מתחתיו מחזיר null (לא זיהה נוסח), כדי לא לסנן בית כנסת בטעות לנוסח שגוי.

שכבת החוקים (Rule-based overrides) — למה הכרחית

המודל לא מושלם, ויש לו טעויות ידועות ושיטתיות. אחרי הסיווג יש בדיקות תיקון:

  • בעיית "בית": ערים כמו "בית שמש", "בית שאן" מכילות "בית" → המודל נוטה לסווג כ-synagogue. containsFoodTerm מזהה מילת אוכל ("פיצה", "שווארמה") ומכריח restaurant.
  • "בתי כנסת" מפורש → containsSynagogueExplicitTerm מכריח synagogue.
  • אירוח: containsHostingSignal — "אירוח", או "שבת"+"להתארח".
זו מערכת היברידית — ML לרוב המקרים, ושכבת חוקים דקה שמתקנת טעויות ידועות. זה לא 'רמאות' — בתעשייה מערכות production כמעט תמיד משלבות ML עם heuristics. ה-ML נותן גמישות, החוקים נותנים דיוק בנקודות הרגישות.

חסרונות ומגבלות המודל (תהיה כן!)

  1. דאטה קטן — 728 דוגמאות זה מעט. מודל מתקשה עם ניסוחים שונים מאוד מאלה שראה.
  2. אנגלית חלשה — רוב הדאטה עברית; באנגלית הביצועים נמוכים יותר.
