# חוברת הכנה להגנת פרויקט גמר

פרויקט: `Jewish On The Way`  
מטרה: מסמך קריאה והדפסה שמרכז את ההסברים, הביקורת, השאלות והתשובות שעלו בהכנה להגנה.

---

## 1. תקציר הפרויקט

`Jewish On The Way` היא אפליקציית מובייל שמרכזת שירותים יהודיים לפי יעד ומיקום: מסעדות כשרות, בתי כנסת, מניינים, אירוח שבת, מועדפים, ביקורות וצ'אט.

החלק המרכזי בפרויקט הוא החיפוש החכם. המשתמש יכול לכתוב משפט חופשי כמו:

```text
פיצה במזכרת בתיה
בית כנסת קרוב
בית כנסת בדובאי
מניין שחרית ספרדי בירושלים
אירוח שבת בפוקט
```

והמערכת מבינה:

- איזו קטגוריה המשתמש מחפש.
- האם יש יעד מפורש בטקסט.
- האם להשתמש ב-GPS.
- האם צריך פילטר כמו כשרות, סוג מסעדה או נוסח.
- איך להחזיר תוצאות רלוונטיות לפי עיר, מרחק ודירוג.

משפט קצר להגנה:

> הפרויקט הוא אפליקציית מובייל למציאת שירותים יהודיים לפי טקסט חופשי, יעד ומיקום. החיפוש החכם משלב מודל AI מקומי, חוקים ידניים, זיהוי יעדים, GPS, PostGIS ודירוג תוצאות.

## 2. הטכנולוגיות המרכזיות בפרויקט

### TypeScript

TypeScript היא שפה שמבוססת על JavaScript ומוסיפה לה טיפוסים. בסוף הקוד מתקמפל ל-JavaScript, אבל בזמן הפיתוח TypeScript עוזרת לזהות טעויות מראש.

למה השתמשתי בזה:

- הפרויקט כולל הרבה סוגי נתונים: משתמשים, מסעדות, בתי כנסת, מניינים, יעדים, ביקורות וחיפושים.
- טיפוסים עוזרים להבין מה כל פונקציה מקבלת ומה היא מחזירה.
- NestJS ו-React Native עובדים טוב מאוד עם TypeScript.
- זה מפחית טעויות בזמן פיתוח.

תשובה להגנה:

> בחרתי TypeScript כי הפרויקט גדול ומכיל הרבה ישויות ומודולים. TypeScript מאפשר להגדיר טיפוסים ברורים, לזהות טעויות לפני הרצה, ולשמור על קוד מסודר גם בצד השרת וגם בצד המובייל.

### JavaScript

JavaScript היא שפת התכנות שעליה TypeScript מבוססת. בפועל, אחרי קומפילציה, הקוד שרץ הוא JavaScript.

### Node.js

Node.js היא סביבת הרצה שמאפשרת להריץ JavaScript בצד שרת. ה-backend של הפרויקט רץ על Node.js.

תשובה להגנה:

> Node.js מאפשר להריץ JavaScript/TypeScript בצד שרת. בפרויקט שלי הוא משמש להרצת השרת שמקבל בקשות מהמובייל, מדבר עם מסד הנתונים, מבצע חיפוש, התחברות, ביקורות וצ'אט.

### Backend

Backend הוא צד השרת. הוא אחראי על:

- התחברות והרשאות.
- שמירת נתונים במסד הנתונים.
- חיפוש חכם.
- החזרת מסעדות, בתי כנסת, מניינים ואירוח.
- צ'אט בזמן אמת.
- ביקורות, דיווחים ומועדפים.

### Frontend / Mobile

Frontend הוא הצד שהמשתמש רואה. בפרויקט זה אפליקציית מובייל שנבנתה עם React Native ו-Expo.

### API

API הוא דרך תקשורת בין האפליקציה לשרת. לדוגמה:

```text
POST /search
GET /restaurants/:destinationId
GET /synagogues/:destinationId
POST /auth/login
```

תשובה להגנה:

> ה-API הוא החוזה בין המובייל לשרת. המובייל שולח בקשות, והשרת מחזיר נתונים או מבצע פעולות.

### REST API

REST API הוא סגנון נפוץ לבניית endpoints. משתמשים ב-HTTP methods כמו `GET`, `POST`, `PUT`, `DELETE`.

### NestJS

NestJS היא framework לבניית שרתים ב-Node.js עם TypeScript. היא מחלקת את הקוד ל-modules, controllers, services, guards ו-DTOs.

למה השתמשתי בזה:

- מבנה מסודר לפרויקט גדול.
- הפרדה בין קבלת בקשה לבין לוגיקה עסקית.
- תמיכה טבעית ב-TypeScript.
- נוח לאבטחה, בדיקות ותחזוקה.

### Controller

Controller מקבל בקשות HTTP. לדוגמה, אם האפליקציה מבקשת מסעדות, controller מקבל את הבקשה ומעביר ל-service.

### Service

Service מכיל את הלוגיקה העסקית. למשל איך לחפש מסעדות, איך להשלים בתי כנסת קרובים, איך לבצע login.

### Module

Module מרכז תחום מסוים. לדוגמה:

```text
RestaurantsModule
SynagoguesModule
AuthModule
ChatModule
ReviewsModule
FavoritesModule
```

### DTO

DTO מגדיר איזה מידע מותר לקבל בבקשה. זה עוזר לוולידציה ולסדר.

### Guard

Guard הוא שכבת הגנה. לדוגמה:

- `JwtAuthGuard` בודק אם המשתמש מחובר.
- `AdminGuard` בודק אם המשתמש מנהל.

### React Native

React Native מאפשרת לבנות אפליקציית מובייל עם TypeScript/JavaScript. במקום לכתוב אפליקציה נפרדת לאנדרואיד ולאייפון, כותבים בסיס קוד אחד.

### Expo

Expo מקל על פיתוח React Native ומאפשר להריץ את האפליקציה דרך Expo Go בזמן פיתוח.

חשוב לדעת:

> Expo Go מצוין לפיתוח, אבל בהגנה צריך לוודא מראש שהשרת רץ, שהטלפון מחובר, ושהאפליקציה נטענת. לסביבה סופית עדיף build מסודר.

### PostgreSQL

PostgreSQL הוא מסד נתונים רלציוני. הוא שומר טבלאות כמו משתמשים, יעדים, מסעדות, בתי כנסת, ביקורות ועוד.

למה בחרתי:

- יציב וחזק.
- מתאים לקשרים בין טבלאות.
- תומך ב-PostGIS.

### TypeORM

TypeORM מחבר בין TypeScript לבין PostgreSQL. הוא מאפשר לעבוד עם entities ו-repositories במקום לכתוב SQL ידני בכל מקום.

### Entity

Entity היא מחלקה שמייצגת טבלה במסד הנתונים. לדוגמה `User`, `Restaurant`, `Destination`.

### Repository

Repository מאפשר לבצע פעולות מול טבלה: חיפוש, שמירה, עדכון ומחיקה.

### PostGIS

PostGIS היא הרחבה ל-PostgreSQL שמאפשרת חישובי מיקום ומרחק.

בפרויקט היא חשובה בשביל:

- מציאת יעד קרוב לפי GPS.
- השלמת תוצאות עד 30 ק"מ.
- מיון תוצאות לפי מרחק.

### JWT

JWT הוא token חתום שהשרת נותן למשתמש אחרי התחברות. האפליקציה שולחת אותו בכל בקשה כדי להוכיח שהמשתמש מחובר.

### Access Token ו-Refresh Token

Access token הוא token קצר חיים לבקשות רגילות.  
Refresh token מאפשר לקבל access token חדש בלי להתחבר מחדש.

בפרויקט refresh token נשמר בשרת כ-hash עם SHA-256, לא כטקסט רגיל.

### bcrypt

bcrypt משמש לשמירת סיסמאות בצורה חד-כיוונית. לא שומרים סיסמה רגילה במסד הנתונים, אלא hash.

### SecureStore ו-AsyncStorage

במובייל:

- `SecureStore` שומר tokens בצורה מאובטחת במכשיר.
- `AsyncStorage` משמש לדברים מקומיים כמו מועדפים מקומיים, שפה, cache ועוד.

### WebSocket / Socket.IO

WebSocket מאפשר תקשורת בזמן אמת. בפרויקט יש צ'אט בזמן אמת לערים, מניינים ואירוח.

יש:

- אימות JWT בסוקט.
- rooms.
- typing.
- online presence.
- read cursors.
- rate limiting להודעות.

## 6. TF-IDF

TF-IDF היא שיטה להפוך טקסט למספרים לפי חשיבות מילים.

TF = Term Frequency  
כמה מילה מופיעה במשפט.

IDF = Inverse Document Frequency  
כמה המילה נדירה בכלל הדאטה.

TF-IDF נותן משקל גבוה יותר למילים שעוזרות להבדיל בין קטגוריות.

דוגמה:

- "מסעדה" עוזרת לזהות restaurant.
- "מניין" עוזרת לזהות minyan.
- "אירוח" עוזרת לזהות hosting.
- "בית כנסת" עוזרת לזהות synagogue.

תשובה להגנה:

> TF-IDF הופך משפט לווקטור מספרי לפי חשיבות המילים. מילה שמופיעה במשפט וגם יחסית נדירה בדאטה מקבלת משקל גבוה יותר, כי היא עוזרת להבין את הכוונה.

## 7. Naive Bayes

Naive Bayes הוא אלגוריתם סיווג הסתברותי.

הוא שואל:

> בהינתן המילים במשפט, איזו קטגוריה הכי סבירה?

לדוגמה:

```text
פיצה כשרה בתל אביב -> restaurant
מניין שחרית ספרדי -> minyan
אירוח שבת בפוקט -> hosting
בית כנסת בדובאי -> synagogue
```

תשובה להגנה:

> Naive Bayes מחשב לכל קטגוריה ציון הסתברותי לפי המילים שהופיעו במשפט, ובוחר את הקטגוריה עם הציון הגבוה ביותר. הוא מתאים לסיווג טקסט קצר כי הוא מהיר, פשוט וניתן להסבר.

## 8. Bigram

במודל הראשי יש גם bigrams, כלומר צמדי מילים.

דוגמה:

```text
בית כנסת -> בית__כנסת
אירוח שבת -> אירוח__שבת
מסעדה כשרה -> מסעדה__כשרה
```

למה זה חשוב:

- "בית" לבד יכול להיות חלק מ"בית שמש".
- "בית כנסת" ביחד הוא סימן ברור לבית כנסת.

תשובה להגנה:

> הוספנו bigrams כדי שהמודל יבין ביטויים, לא רק מילים בודדות. זה חשוב במיוחד בעברית, שבה צירופי מילים משנים משמעות.

## 12. שכבת החוקים מעל המודל

המודל לא עובד לבד. מעליו יש חוקים שמתקנים מקרי קצה.

דוגמאות:

- מילים כמו `פיצה`, `burger`, `שווארמה` מכריחות `restaurant`.
- הביטוי `בית כנסת` מכריח `synagogue`.
- מילים כמו `אירוח`, `להתארח`, `hosting` מכריחות `hosting`.


המודל לבדו עלול לסווג לא נכון, אבל שכבת החוקים מזהה `burger` כמילת אוכל ומתקנת למסעדה.

תשובה להגנה:

> לא הסתמכתי רק על המודל כי בפרויקט אמיתי חשוב לצמצם טעויות. לכן בניתי מנוע היברידי: המודל נותן הבנה כללית, והחוקים מתקנים מקרי קצה חשובים.

## 13. זיהוי יעד

המודל הראשי לא אחראי לזהות יעדים. בשביל זה יש שכבת `DestinationIndexService`.

היא עושה:

- טעינת יעדים מה-DB.
- בניית aliases בעברית ובאנגלית.
- ניקוי טקסט.
- יצירת candidates.
- סינון stop words.
- fuzzy matching עם Levenshtein.

דוגמאות:

```text
מזכרת בתיה -> Mazkeret Batya
דובאי -> Dubai
פוקט -> Phuket
בית שמש -> Beit Shemesh
```

תשובה להגנה:

> זיהוי יעד לא מתבצע דרך המודל הראשי אלא דרך אינדקס יעדים. המערכת מחזיקה aliases בעברית ובאנגלית, מייצרת מועמדים מתוך הטקסט, ומבצעת התאמה מדויקת או fuzzy matching.

## 14. GPS מול יעד מפורש

זה אחד הדברים הכי חשובים בפרויקט.

אם המשתמש כותב:

```text
בית כנסת קרוב
```

ואין יעד מפורש, המערכת משתמשת ב-GPS כדי למצוא יעד קרוב.

## 20. Precision, Recall ו-F1

Precision:

> מתוך כל הפעמים שהמודל אמר קטגוריה מסוימת, כמה פעמים הוא צדק.

Recall:

> מתוך כל הדוגמאות האמיתיות של קטגוריה מסוימת, כמה המודל הצליח למצוא.

F1:

> מדד שמאזן בין Precision ו-Recall.

תשובה להגנה:

> Accuracy נותן תמונה כללית, אבל Precision, Recall ו-F1 מראים איפה המודל חזק ואיפה הוא חלש בכל קטגוריה.

## 22. בעיית test_classifier.py והדיוק

הייתה נקודה חשובה:

`classifier.py` מודד דיוק אמיתי עם train/test split.

ההסבר הנכון:

> יש הבדל בין training accuracy לבין held-out accuracy. בדיקה שמאמנת ובודקת על אותו דאטה היא sanity check, והיא רק אומרת שהמודל למד את הדוגמאות. המדד האמיתי הוא held-out test, שבו המודל נבדק על משפטים שלא ראה באימון.

## 28. תשובות מוכנות לשאלות נפוצות

### מה זה TypeScript ולמה השתמשת בו?

TypeScript היא שפה שמבוססת על JavaScript ומוסיפה טיפוסים. השתמשתי בה כי הפרויקט גדול ומכיל הרבה ישויות. הטיפוסים עוזרים למנוע טעויות ולשמור על קוד מסודר.

### מה זה Node.js?

Node.js היא סביבת הרצה שמאפשרת להריץ JavaScript בצד שרת. השרת שלי רץ על Node.js דרך NestJS.

### מה זה NestJS?

NestJS היא framework ל-backend ב-TypeScript. היא נותנת מבנה של modules, controllers, services ו-guards.

### למה לא Firebase?

רציתי שליטה מלאה בלוגיקה מורכבת: חיפוש חכם, PostGIS, דירוג תוצאות, roles, imports, ביקורות וצ'אט. שרת עצמאי מתאים יותר לפרויקט כזה.

### למה PostgreSQL?

כי הנתונים רלציוניים ויש הרבה קשרים בין טבלאות. בנוסף PostgreSQL תומך ב-PostGIS לחישובי מרחק.

### מה זה PostGIS?

PostGIS היא הרחבה ל-PostgreSQL שמאפשרת עבודה עם מיקומים גיאוגרפיים וחישובי מרחקים.

### מה זה JWT?

JWT הוא token חתום שמייצג משתמש מחובר. האפליקציה שולחת אותו לשרת בכל בקשה שדורשת הרשאה.

### מה זה refresh token?

refresh token מאפשר לקבל access token חדש בלי שהמשתמש יתחבר מחדש. בפרויקט הוא נשמר בשרת כ-hash.

### מה זה bcrypt?

bcrypt משמש לשמירת סיסמאות בצורה חד-כיוונית. לא שומרים סיסמה רגילה במסד הנתונים.

### מה זה WebSocket?

WebSocket מאפשר תקשורת דו-כיוונית בזמן אמת. בפרויקט הוא משמש לצ'אט ונוכחות.

### מה זה TF-IDF?

שיטה להפוך טקסט למספרים לפי חשיבות המילים.

### מה זה Naive Bayes?

אלגוריתם שמחשב איזו קטגוריה הכי סבירה לפי המילים במשפט.

### למה לא השתמשת ב-ChatGPT?

כי רציתי מודל מקומי, מהיר, זול, ניתן להסבר ולא תלוי בשירות חיצוני.

### מה החיסרון של המודל שלך?

הוא פחות מבין שפה מורכבת ממודל גדול ותלוי בדאטה. לכן שילבתי אותו עם חוקים ו-fallbacks.

### מה היתרון של המודל שלך?

הוא מהיר, מקומי, לא עולה כסף לכל בקשה, לא תלוי ב-API חיצוני, וניתן להסביר אותו לעומק.

### מה האתגר הכי גדול?

לגרום לטקסט חופשי להפוך לתוצאות נכונות לפי קטגוריה, יעד, מיקום, מרחק ורלוונטיות.

### מה היית משפר בעתיד?

עוד דאטה, hard negatives, confusion matrix, בדיקות אוטומטיות, HTTPS מלא בפרודקשן, ואולי embeddings בהמשך.
